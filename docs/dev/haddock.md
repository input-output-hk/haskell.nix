# Building haddock

Under the **v1** builder, every component derivation exposed a
`.doc` attribute generated by a sibling haddock derivation that
shared the lib's UnitId.  v1's `Setup configure` could be re-run
with the same args and `Setup haddock` invoked against the same
build tree, so the lib's UnitId was unaffected by whether haddock
was generated.

Under the **v2** builder this no longer holds: `cabal v2-haddock`
includes `--enable-documentation` (and the propagated
`elabBuildHaddocks` field) in `pkgHashConfigInputs`, so a
haddock-generating build of a package always produces a different
UnitId than a non-haddock build.  Calling `cabal v2-haddock`
against a plan that doesn't already have `--enable-documentation`
forks every UnitId in the build closure and triggers a fresh
re-build of every module.

## How to expose `.doc` on a v2 slice

Add to `cabal.project` (or `cabalProjectLocal`):

```cabal
package *
  documentation: True
```

(or `package <pkg-name>` to enable docs for a single package).
That setting lands in plan.json's `configure-args` as
`--enable-documentation`, which:

* feeds the haddock-aware UnitId into plan-nix from the start, so
  the main slice's `cabal v2-build` and the doc slice's
  `cabal v2-haddock` agree on the UnitId; and
* lets `slice.doc` resolve to a sibling derivation that runs
  haddock against the already-installed unit (no closure rebuild).

Without that setting, attribute access to `slice.doc` throws with
a message pointing at this document.

## What's in `slice.doc`'s `$out`

The doc slice keeps cabal's native unit-dir layout:

```
$out/store/<ghc>/<unit-id>/share/doc/html/[<sublib>/]…
```

The same shape as `$out/store/<ghc>/<unit-id>/lib/…` for the rest
of the unit's contents.  This means the doc slice can be
`lndir`'d into `~/.cabal/store/<ghc>/<unit-id>/` as a drop-in
replacement for the corresponding non-doc slice — cabal-install
will see the haddock-html / haddock-interfaces fields populated
in the unit's `.conf` and use them for cross-package hyperlinks.

Cross-package hyperlinks in the generated html are absolute
`file:///nix/store/<doc-slice>/store/<ghc>/<dep-unit-id>/share/doc/html/…`
URLs (haddock embeds the build-time `--read-interface=` paths).
They resolve back into the doc slice's own `$out/store/` tree on
the same host.

Each doc slice carries only its own target's html in
`$out/store/`; deps' haddocks land in the deps' own `.doc`
slices, which the doc slice's `depSlices` pull in via
`(map (d: d.doc) depSlices)`.  This keeps each slice lean while
making the cross-package hyperlinks resolve.

## Why no `slice.haddockDir`

For local packages, plan-nix records the
`<pkg>-<ver>-inplace[-<comp>]` UnitId form, but the slice's
cabal store actually uses cabal's mangled hashed form
(`<short>-<ver>-<8-char-hash>`).  There is no eval-time-stable
path to the html.  Callers that need the html should either
`find` it under the doc slice, or use the doc slice as a whole
(e.g. `nix-build -A …components.library.doc`).
